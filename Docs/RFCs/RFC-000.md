# RFC-000 — Kalsae vs Tauri vs Wails 보안 영역 비교 프레임워크

| 항목 | 내용 |
|------|------|
| 상태 | 초안 (Draft, rev.2) |
| 날짜 | 2026-05-16 |
| 영향 범위 | 문서 전용 — 소스 코드 변경 없음 |
| 비교 기준 버전 | **Kalsae** 0.3.x (main, 2026-05-16) · **Tauri** v2.x stable · **Wails** v3.0.0-alpha (multi-window·drop handler·packaging만 부분 평가) |
| 관련 | `Docs/SECURITY.md`, `Docs/ARCHITECTURE.md`, `Docs/IPC.md`, `Docs/LAYERS.md`, `Sources/KalsaeCore/Config/KSSecurityConfig.swift`, `Sources/KalsaeCore/Config/Capabilities/` |

---

## 1. 동기 (Motivation)

Kalsae는 Tauri(Rust)의 보안 모델에서 영감을 받아 **default-deny** 철학을 채택했으나,
Wails(Go)와 비교해 어떤 영역이 강점이고 어떤 영역이 개선 여지가 있는지 체계적으로
분석된 자료가 없다.

세 프레임워크를 동일한 기준으로 비교할 수 있는 영역 분류가 필요하다. 본 RFC는
이를 위한 **15대 보안 비교 영역**을 정의하고, 각 영역별로 Kalsae의 현재 위치와
개선 우선순위를 도출하는 프레임워크를 제공한다.

> **범위 한정**: 본 RFC는 **보안 축만** 다룬다. 기능 · 개발자 경험(DX) ·
> 빌드/배포 · 성능 · 플랫폼 커버리지 비교는 별도 RFC 시리즈에서 다룬다.
> 따라서 종합 점수도 "보안 관점의 상대 평가"이지 프레임워크 우열의 단일 척도가
> 아니다.

---

## 2. 목표 / 비목표

### 목표

- Kalsae · Tauri(v2) · Wails(v3-alpha)의 보안 기능을 동일 선상에서 비교할 수 있는
  15개 영역을 정의한다.
- 각 영역별로 Kalsae의 현재 구현 수준을 평가하고, Tauri 대비 격차(gap)를
  수치화한다.
- 개선 우선순위(🔴高 / 🟡中 / 🟢低)를 제시하여 로드맵 결정에 활용할 수 있게 한다.

### 비목표

- 실제 PoC 코드나 성능 벤치마크 제공 — 본 RFC는 분석 프레임워크만 정의한다.
- Tauri나 Wails의 내부 구현을 완전히 재현 — 공식 문서와 공개된 API 시그니처를
  기준으로 비교한다. 특히 **Wails v3는 alpha 단계**이므로 일부 영역은 alpha
  시점 스냅샷이며, stable 릴리스 시 재평가가 필요하다.
- 소스 코드 변경 — 본 RFC는 `Docs/` 전용 문서이며, breaking change를 동반하는
  구체 작업은 별도 RFC(RFC-002 등)에서 다룬다.
- 기능 · DX · 빌드 · 성능 · 플랫폼 비교 — 별도 RFC 시리즈에서 다룬다.

## 3. 15대 보안 비교 영역

### 3.1 영역 분류 기준

다음 세 가지 축을 기준으로 영역을 나눴다:

1. **공격 표면(Attack Surface)**: JS → Native 경로(IPC), WebView ↔ OS 경계,
   파일/네트워크/셸 접근
2. **방어 계층(Defense Layer)**: 언어 수준(컴파일 타임), 런타임(인가/검증),
   설정(정책 선언)
3. **프레임워크 특성(Framework Nature)**: 각 프레임워크가 특별히 강조하거나
   부재한 보안 기능

| # | 영역 | 공격 표면 | 방어 계층 | 비교 가중치 |
|---|------|-----------|-----------|------------|
| 1 | 아키텍처 보안 | 언어/런타임 | 컴파일 타임 | 🟢 기본 — 언어 선택의 결과 |
| 2 | IPC 보안 | JS→Native | 런타임 + 설정 | 🔴 가장 중요 — 모든 명령의 관문 |
| 3 | WebView 경계 보안 | WebView↔OS | 런타임 + 설정 | 🔴 가장 중요 — 브라우저 취약점 차단 |
| 4 | 파일시스템 접근 제어 | 디스크 I/O | 런타임 + 설정 | 🔴 가장 중요 — 데이터 유출 방지 |
| 5 | HTTP/네트워크 통신 제어 | 네트워크 I/O | 런타임 + 설정 | 🟡 중요 — CORS 우회 방지 |
| 6 | 셸 통합 보안 | OS 명령 실행 | 런타임 + 설정 | 🟡 중요 — 임의 명령 실행 방지 |
| 7 | 알림 권한 제어 | OS UX | 설정 | 🟢 표면 얕음 |
| 8 | 다중 창 격리 | 창 간 데이터 흐름 | 런타임 | 🟡 중요 — 격리 위반 방지 |
| 9 | 단일 인스턴스 & 딥 링크 | 프로세스 간 | 런타임 | 🟢 표면 얕음 |
| 10 | 공급망 보안 | 설정/빌드 | 컴파일 타임 + 설정 | 🟡 중요 — 릴리스 파이프라인 |
| 11 | 보안 관찰 가능성 | 모니터링 | 런타임 | 🟢 디버깅/감사 |
| 12 | 보안 설정 표현 및 UX | 설정 UX | 설정 | 🟢 사용자 경험 |
| 13 | 민감 정보/Secret 저장 | OS 보안 저장소 | 런타임 + 설정 | 🟡 중요 — 자격 증명 보호 |
| 14 | WebView 프로세스 격리 | 프로세스 모델 | 런타임 | 🟡 중요 — 샌드박스 경계 |
| 15 | 자동 업데이트 보안 | 업데이트 파이프라인 | 컴파일 타임 + 설정 | 🔴 가장 중요 — 코드 무결성 |

> **점수 체계**: 본 RFC는 각 영역을 **단순 ⭐1~5로 평가하고 산술 합산**한다(만점 15×5 = 75).
> 위 가중치 열은 **우선순위 표시용 정성 라벨**이며 점수 합산에는 반영되지 않는다.
> 가중 합산 도입 여부는 §7 미결 사항 참조.

### 3.2 영역 1: 아키텍처 보안 (Architectural Security)

선택한 언어/런타임이 근본적으로 제공하는 메모리 안전성과 동시성 안전.

| 항목 | Kalsae (Swift 6) | Tauri (Rust) | Wails (Go) |
|------|-------------------|--------------|------------|
| **메모리 안전성** | ARC (컴파일 타임 + 런타임 참조 카운팅) | **소유권/대여 시스템** (컴파일 타임 완전 보장, `unsafe` opt-in) | GC (런타임 추적, 순환 참조는 GC에 의존) |
| **데이터 경합 방지** | `actor` + `@Sendable` + `MainActor` (컴파일 타임 격리) | `Send + Sync` 트레이트 바운드 (컴파일 타임 검증) | goroutine + channel (런타임, `-race` 플래그로만 탐지) |
| **에러 처리** | `throws(KSError)` typed throws (컴파일 타임 에러 타입 추론) | `Result<T, E>` + `match` (철저한 패턴 매칭 강제) | `error` 인터페이스 + `if err != nil` (덜 엄격) |
| **널 안전성** | `Optional<T>` (컴파일 타임, `!`는 금지됨) | `Option<T>` (컴파일 타임, `unwrap()`은 명시적) | `nil` 포인터 가능 (Go 1.24+ 제네릭 `interface{T}`에서 완화 추세) |

**평가**: Rust > Swift > Go 순. Kalsae는 typed throws와 actor 모델로 Tauri에 근접하나,
Rust의 소유권 시스템이 메모리 안전에서 본질적으로 우위.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐ | ARC + actor + typed throws, 단 force unwrap만 없으면 완벽 |
| Tauri | ⭐⭐⭐⭐⭐ | 소유권 + Send/Sync + Result, 컴파일 타임 안전의 최정점 |
| Wails | ⭐⭐⭐ | GC + goroutine, 런타임 비용과 데이터 경합 위험 |

**Kalsae 개선 필요성**: 🟢 당장 개선 필요한 부분 없음 (언어 선택의 결과)

### 3.3 영역 2: IPC 보안 (명령 호출 경로)

JS → Native 명령 호출의 인증/인가/검증. **가장 중요한 공격 표면**.

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **명령 허용 목록** | `commandAllowlist` (`[String]?`)<br>`nil` → **등록된 non-internal 명령 전체**(기본 집합)<br>`[]` → 모두 거부<br>**+ Tauri 스타일 `capabilities` 시스템 별도 도입**(`Sources/KalsaeCore/Config/Capabilities/`) | Capabilities + Permissions 시스템<br>**기본값: deny-zero (아무것도 허용 안 함)** | **없음**<br>모든 `func` binding이 JS에 자동 노출 |
| **세분화된 권한** | `KSPermission` + `KSCapability` (identifier, commandsAllow/Deny, scope) — Tauri 모델과 동등한 표현력 | Permission + Role 계층 (`allow`, `deny`, `require` 등) | N/A |
| **속도 제한** | `commandRateLimit` (token bucket, `nil`=비활성화) | **없음** (plugin이 자체 구현) | **없음** |
| **IPC 프레임 크기 제한** | 16MB (`KSIPCBridgeCore.maxFrameBytes`) | WRY 레벨에서 WebView2/WebKitGTK/WKWebView 기본값 | **없음** (Go `json.Unmarshal`이 수신) |
| **인자 검증** | Scope 기반 (경로/URL → `security.fs`, `security.navigation`, RFC-002 TOCTOU 방지) | `tauri::command` 타입 검증 + Scope 검증 | Go 타입 검증만 (스키마 없음) |
| **TOCTOU 방지** | 검증 후 표준화된 경로를 PAL에 전달 | Scope 검증 후 `std::fs` 접근까지 동일 scope 내 처리 | N/A |
| **에러 코드** | `commandNotFound`, `commandNotAllowed`, `rateLimited` 등 9개 | `InvokeError` + plugin별 에러 타입 | 일반 `error` 반환 |

**핵심 발견**: Kalsae는 이미 **Tauri v2 스타일 `capabilities` 시스템**을 자체적으로 도입했다
(`KSCapabilitiesConfig`, `KSPermission`, `KSCapability`, `KSPermissionScope`). 다만
**기본 진입점인 `security.commandAllowlist`의 의미론이 여전히 permissive**하다 —
`nil`(미설정)이 "등록된 명령 전체 자동 허용"으로 해석되므로 Tauri의 deny-zero보다 약하다.
`capabilities.*`를 선언하면 사실상 Tauri와 동급의 권한 그래프를 얻지만,
**기존 사용자 경로**는 여전히 permissive default라는 점이 핵심 갭이다.

```json
// Kalsae 현재: commandAllowlist가 없으면 등록된 명령 전체가 자동 허용
// Kalsae.json
{ "security": { /* commandAllowlist 없음 → 기본 집합(non-internal 전체) */ } }

// Kalsae 권장(이미 가능): capabilities 명시
{
  "capabilities": {
    "permissions": [{ "identifier": "fs:read", "commandsAllow": ["fs.read*"] }],
    "capabilities": [{ "identifier": "main", "windows": ["main"], "permissions": ["fs:read"] }]
  }
}

// Tauri v2: capabilities를 설정하지 않으면 어떤 명령도 호출 불가
// capabilities/default.json
// ← 파일이 없으면 모든 IPC 호출이 거부됨
```

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐ | Capabilities 시스템 구현 완료. 단 레거시 `commandAllowlist` 기본값이 permissive |
| Tauri | ⭐⭐⭐⭐⭐ | Capabilities 모델 + deny-zero 기본값, 업계 최고 수준 |
| Wails | ⭐⭐ | binding 보안 메커니즘 자체가 없음 |

**Kalsae 개선 필요성**: 🔴 **최우선 개선 대상**
- `commandAllowlist == nil`을 "deny-all"로 재해석(또는 `capabilities`를 단일 진입점으로 승격)
- 이미 `capabilities` 시스템이 존재하므로 **새 기능 추가가 아닌 default 변경**이 핵심
- 단, 이는 **breaking change**이므로 RFC-002에서 마이그레이션 전략과 함께 제안

### 3.4 영역 3: WebView 경계 보안

브라우저 엔진과 네이티브 호스트 사이의 경계 통제.

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **CSP 주입** | ✅ HTTP header (Windows WebResourceRequested)<br>✅ Meta tag (그 외 모든 플랫폼) | ✅ WRY의 `WebviewBuilder`에서 설정 가능<br>✅ 매 요청마다 헤더 주입 | ❌ **자체 CSP 관리 없음**<br>사용자가 HTML `<meta>`로 직접 |
| **네비게이션 제어** | ✅ `navigation.allow` + `openInBrowser` | ✅ `WebviewWindowBuilder.navigation_scope()`<br>(더 세분화된 scope 제어) | ❌ 없음<br>`SetURL()`만 존재 |
| **팝업 차단** | ✅ `allowPopups: false` (기본 차단) | ✅ `WindowBuilder._popup_handler()` private API | ❌ 기본 WebView 설정에 의존 |
| **DevTools 통제** | ✅ 릴리스 빌드 강제 비활성화<br>(`#if DEBUG` 외에는 무시) | ✅ Cargo `devtools` feature flag로 **빌드 타임 제거**<br>(릴리스 바이너리에서 코드 자체가 빠짐) | ⚠️ `WebView.Debug` boolean<br>릴리스에서도 개발자가 켜면 열림 |
| **다운로드 통제** | ✅ `downloads.enabled: false` (기본 차단) | ✅ `WebviewWindowBuilder.download_handler()`<br>다운로드 시작/차단 결정 | ❌ 없음 |
| **외부 드롭 제어** | ✅ `allowExternalDrop: false` (기본 차단)<br>→ `__ks.file.drop` 이벤트로 우회 | ✅ `drag_drop_handler()` + `drop_handler()` | ❌ 없음 |
| **컨텍스트 메뉴 제어** | ✅ `contextMenu: .default / .disabled` | ✅ `context_menu_handler()` | ❌ 없음 |
| **교차 오리진 격리** | ✅ `crossOriginIsolation: Bool`<br>→ CoOP/CoEP/CoRP 헤더 주입 | ⚠️ `custom_protocol` + header 조작으로 가능<br>(별도 API 없음) | ❌ 없음 |
| **User-Agent / 기타 헤더** | ✅ CSP + nosniff + referrer-policy | ✅ WRY 레벨에서 설정 가능 | ❌ 없음 |
| **XSS 보호 (IPC)** | ✅ JSON 내 `</script>` 이스케이프<br>✅ Unicode line/paragraph separator 차단 | ✅ WRY 레벨 | ❌ 없음 |

**평가**: Kalsae가 이 영역에서 Tauri와 **동등**하다. Wails는 WebView 경계 보안 기능이
거의 전무하다. 이는 Kalsae가 Wails보다 뒤늦게 설계되면서 Tauri의 보안 모델을
참고했기 때문으로 분석된다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐⭐ | CSP, 네비게이션, 드롭, 다운로드, 팝업, DevTools 전방위 통제 |
| Tauri | ⭐⭐⭐⭐⭐ | WRY 레이어에서 동등한 기능 제공, cross-origin 격리만 약간 부족 |
| Wails | ⭐⭐ | WebView 기본값에만 의존 |

**Kalsae 개선 필요성**: 🟢 이미 Tauri 수준. 유지보수만 지속.

### 3.5 영역 4: 파일시스템 접근 제어

JS → Native 파일 읽기/쓰기의 권한 통제.

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **경로 제어 방식** | Allow/deny glob 패턴<br>4개 플레이스홀더: `$APP`, `$HOME`, `$DOCS`, `$TEMP` | `Scope` 구조체 + `BaseDirectory` 열거형<br>**약 22개 플레이스홀더**: `$AUDIO`, `$CACHE`, `$CONFIG`, `$DATA`, `$LOCALDATA`, `$DESKTOP`, `$DOCUMENT`, `$DOWNLOAD`, `$PICTURE`, `$PUBLIC`, `$VIDEO`, `$RESOURCE`, `$TEMP`, `$APPCONFIG`, `$APPDATA`, `$APPLOCALDATA`, `$APPCACHE`, `$APPLOG`, `$HOME`, `$FONT`, `$EXE`, `$RUNTIME` | ❌ **없음**<br>`runtime.OpenFileDialog` 등 사용자가 직접 검증 |
| **Deny 우선** | ✅ `deny`가 `allow` override | ✅ 동일 | N/A |
| **TOCTOU 방지** | ✅ 표준화된 확장 경로를 PAL에 전달 | ✅ Scope 검증 후 동일 scope 내에서 `std::fs` 접근 | ❌ 없음 |
| **파일 메타데이터** | `__ks.fs.metadata` (size, mtime, kind) | `fs:allow-stat` (더 풍부한 메타데이터) | ❌ 없음 |
| **디렉터리 순회** | `__ks.fs.readDir` | `fs:allow-read-dir` | ❌ 없음 |
| **스트리밍 읽기** | 전체 읽기 only (`readTextFile`, `readFile`) | `fs:allow-read-file` (전체) + 스트림 (일부) | ❌ 없음 |

**핵심 발견**: Tauri는 macOS 샌드박스 컨테이너 경로부터 리눅스 XDG 기반 경로까지
약 22개의 `BaseDirectory`를 제공하는 반면, Kalsae는 4개만 지원한다. 특히
`$CACHE`, `$CONFIG`, `$DESKTOP`, `$DOWNLOAD`, `$APPCACHE`, `$APPLOG`는 실제 앱에서
가장 자주 필요한 경로이므로 추가 검토 가치가 있다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐ | Allow/deny 패턴 + TOCTOU 방지 구조는 우수하나, 플레이스홀더 다양성 부족 |
| Tauri | ⭐⭐⭐⭐⭐ | 약 22개 BaseDirectory + scope 기반, 업계 최고 |
| Wails | ⭐ | 파일 시스템 접근 제어 메커니즘 자체가 없음 |

**Kalsae 개선 필요성**: 🟡 `$CACHE`, `$CONFIG`, `$DESKTOP`, `$DOWNLOAD` 플레이스홀더 추가
고려. 단, 실제 필요성은 Kalsae 타깃 앱의 사용 패턴 분석 후 결정.

### 3.6 영역 5: HTTP/네트워크 통신 제어

JS에서 외부 네트워크 요청을 보낼 때의 권한 통제.

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **HTTP fetch 제어** | ✅ `http.allow` URL glob 패턴<br>**기본값: `[]` (모든 fetch 차단)** | ✅ `http:default` permission<br>scope 기반 URL 글로빙<br>**기본값: 차단** | ❌ **없음**<br>`net/http` 사용 시 사용자 책임 |
| **외부 URL 열기** | ✅ `shell.openExternalSchemes`<br>**기본값: `["http", "https", "mailto"]`** | ✅ `shell:allow-open`<br>scope + scheme 조합 | ⚠️ `browser.OpenURL()`<br>제한 없음 |
| **HTTP 메서드 제한** | ✅ `KSHTTPScope.methods` (allow 목록, `nil`=전체 허용) | 일부 제한 가능 (plugin 레벨) | N/A |
| **커스텀 헤더 필터링** | ✅ `KSHTTPScope.defaultHeaders` (호스트가 주입할 헤더 선언) — 단, 위험 헤더 차단 목록은 별도 구현 안 됨 | `http:default`에서 일부 헤더 차단 (Host, Cookie 등) | N/A |

**평가**: Kalsae와 Tauri 모두 HTTP fetch를 기본 차단하는 점은 일치. Wails는 이 영역에서
프레임워크 차원의 보호가 전혀 없다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐⭐ | 기본 차단 + URL 글로빙, Tauri와 동등 |
| Tauri | ⭐⭐⭐⭐⭐ | 동등 + 헤더 필터링이 약간 더 세밀 |
| Wails | ⭐ | 프레임워크 차원 네트워크 통제 없음 |

**Kalsae 개선 필요성**: 🟢 추가 개선 불필요.

### 3.7 영역 6: 셸 통합 보안

OS 셸/파일 관리자와 상호작용하는 명령의 보안.

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **openExternal** | `openExternalSchemes` (scheme allowlist)<br>**기본: http, https, mailto** | `shell:allow-open` (scope + scheme)<br>**기본: 차단** | `browser.OpenURL()`<br>**제한 없음** |
| **showItemInFolder** | Boolean + `fsScope` 경로 검증<br>(RFC-002, default-deny) | `shell:allow-show-in-folder` + scope<br>**기본: 차단** | ❌ 없음 |
| **moveToTrash** | Boolean + `fsScope` 경로 검증<br>(RFC-002, default-deny) | `shell:allow-trash` + scope<br>**기본: 차단** | ❌ 없음 |
| **명령 실행 (shell cmd)** | ❌ **없음** (의도적 설계) | `shell:allow-execute` + scope<br>⚠️ 위험하므로 권장 안 함 | ❌ 없음 (사용자가 Go 코드로 직접) |

**평가**: Kalsae는 RFC-002에서 `fsScope` 경로 검증을 추가하면서 Tauri 수준에 도달했다.
단, Tauri가 **기본 차단**인 반면 Kalsae는 `openExternal`이 **3개 scheme allowlist
(`http`, `https`, `mailto`)로 기본 허용**되어 default-deny 철학과 부분적 불일치가 있다.
(`showItemInFolder`, `moveToTrash`는 기본 차단.)

> Tauri는 `openExternal`도 기본 차단(`shell:default`를 명시적으로 추가해야 허용).
> Kalsae는 위 3개 scheme을 기본 허용한다 — '합리적 기본값' 관점에서는
> 방어적이지만 철학적으로는 절충.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐ | RFC-002로 Tauri 수준 도달. 단 `openExternal` 기본 허용은 논란 여지 |
| Tauri | ⭐⭐⭐⭐⭐ | 기본 차단 + scope + permission 계층 |
| Wails | ⭐ | 프레임워크 차원 셸 보안 없음 |

**Kalsae 개선 필요성**: 🟡 `openExternalSchemes` 기본값을 `nil`(제한 없음)이 아닌
현재 `["http", "https", "mailto"]`는 현실적 절충 — 변경 시 사용자 편의성 대폭 하락.

### 3.8 영역 7: 알림 권한 제어

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **알림 post** | `post: true` (기본 허용) | `notification:default` permission<br>**기본: false (차단)** | ❌ 없음 (OS API 직접 호출) |
| **알림 cancel** | `cancel: true` (기본 허용) | 통합 permission | ❌ 없음 |
| **권한 요청** | `requestPermission: true` (기본 허용) | 통합 permission | ❌ 없음 |
| **아이콘 경로 검증** | `iconPath` → `security.fs` 검증 | plugin 레벨에서 처리 | ❌ 없음 |

**핵심 발견**: Kalsae의 `KSNotificationScope`는 모든 필드가 기본 `true`(허용)다.
이는 SECURITY.md에 명시된 **default-deny 철학과 직접적으로 배치**된다.
Tauri는 기본 차단이며, 사용자가 필요한 플러그인만 명시적으로 추가한다.

> **단서**: macOS/iOS `UNNotificationCenter`, Windows 토스트 권한, Linux D-Bus 등
> **OS 자체 권한 프롬프트**가 한 번 더 게이트킵하므로 실질적 피해 가능성은 제한적이다.
> 그러나 "프레임워크 차원에서 default-deny" 원칙 자체를 깨는 것은 일관성 문제를 남긴다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐ | 기능은 세분화되어 있으나 기본값이 default-deny 위배 |
| Tauri | ⭐⭐⭐⭐⭐ | 기본 차단 + plugin 기반 opt-in |
| Wails | ⭐ | 프레임워크 차원 알림 권한 없음 |

**Kalsae 개선 필요성**: 🔴 **두 번째 최우선 개선 대상**
- `KSNotificationScope` 기본값을 `post: false, cancel: false, requestPermission: false`로 변경
- breaking change이므로 마이그레이션 가이드와 함께 RFC에서 제안

### 3.9 영역 8: 다중 창 격리

| 항목 | Kalsae (v0.3+) | Tauri (v2) | Wails (v3-alpha)¹ |
|------|----------------|-------------|-------|
| **창 생성 제어** | ✅ `navigation.allow`로 URL 검증 | ✅ `webview:allow-create-webview-window`<br>(capability 기반) | ⚠️ `application.NewWebviewWindow()` 지원<br>(별도 보안 정책 없음) |

**평가**: Kalsae의 멀티윈도우는 최신 기능(v0.3+)으로 Tauri와 유사한 수준이나,
런타임 동적 창 생성 시 보안 설정 누락(CSP, 가상호스트)이 차이가 있다.
Tauri는 새 창마다 WebviewWindowBuilder를 통해 보안 설정을 재적용한다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐ | 기본 격리 모델은 우수하나 동적 창 생성 시 보안 설정 갭 |
| Tauri | ⭐⭐⭐⭐⭐ | 창 생성 시 모든 보안 설정 재적용 |
| Wails | ⭐⭐ | v3-alpha에서 multi-window 추가됐으나 격리/scope 없음 |

**Kalsae 개선 필요성**: 🟡 `__ks.window.create` 경로에도 CSP/가상호스트/보안정책
재적용 로직 추가 고려. 단, 이는 현재 명시된 limitation(LIMITATIONS.md? 에서
"new window inherits structural settings only")이므로 RFC-002에서 깊이 다룰 필요 있음.

| 항목 | Kalsae | Tauri (v2) | Wails |
|------|--------|-------------|-------|
| **설정 형식** | `Kalsae.json` → `security` 객체<br>계층적 (fs / shell / http / notification 등) | `tauri.conf.json` → `capabilities`<br>+ 별도 `capabilities/` 디렉터리<br>+ `permissions` / `roles` 체계 | `app.go` 구조체 필드<br>minimal |
| **scope 표현** | 직접 경로 패턴 + 플레이스홀더<br>`"allow": ["$APP/data/**"]` | `BaseDirectory` + `Path`<br>`"allow": [{ "path": "$APPDATA/**" }]` | N/A |
| **설정 검증** | ✅ `Decodable` + 런타임 검증<br>✅ `decodeIfPresent`로 하위 호환성 유지 | ✅ JSON 스키마 검증<br>✅ tauri CLI에서 `tauri init` 시 검증 | ❌ Go 구조체 zero value 의존 |
| **문서화** | ✅ `SECURITY.md` 상세함<br>✅ `KSSecurityConfig.swift` 주석 완비 | ✅ 공식 문서 우수<br>✅ API 문서 자동 생성 | ❌ 최소한 |
| **기본값 가시성** | ✅ `KSSecurityConfig.init()`에 모든 기본값 명시 | ✅ 문서에 명시, but 소스 코드 분산 | ❌ |

**평가**: Kalsae와 Tauri 모두 우수한 설정 UX를 제공한다. Tauri의 capability/role 체계가
더 정교하지만, Kalsae의 단순한 `security` 객체도 대부분의 사용 사례를 충분히 커버한다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐⭐⭐ | `KSSecurityConfig`의 `Decodable` + `decodeIfPresent` 설계가 견고 |
| Tauri | ⭐⭐⭐⭐⭐ | 더 정교한 capability/role 시스템, 러닝 커브는 더 높음 |
| Wails | ⭐⭐ | 최소한의 설정만 제공 |

**Kalsae 개선 필요성**: 🟢 추가 개선 불필요. 다만 Tauri의 capabilities 참고는
장기적으로 가치 있음 (v2+에서 검토).

### 3.14 영역 13: 민감 정보 / Secret 저장

토큰·자격증명·API 키 등을 OS의 안전한 저장소에 보관하는 메커니즘.

| 항목 | Kalsae | Tauri (v2) | Wails (v3-alpha) |
|------|--------|-------------|------------------|
| **macOS Keychain 통합** | ❌ 내장 없음 (앱이 직접 `Security.framework` 호출) | ✅ `tauri-plugin-stronghold` 또는 `tauri-plugin-keyring` | ❌ 내장 없음 |
| **Windows DPAPI / Credential Manager** | ❌ 내장 없음 (앱이 직접 `CredWrite` 호출) | ✅ keyring plugin이 wincred 백엔드 | ❌ 내장 없음 |
| **Linux Secret Service / libsecret** | ❌ 내장 없음 | ✅ keyring plugin이 dbus secret-service 백엔드 | ❌ |
| **별도 암호화 vault** | ❌ | ✅ `stronghold` 플러그인 (Iota 기반, 메모리 sealing) | ❌ |
| **JS에서 접근 가능한 API** | ❌ — 직접 PAL 호출 불가 | ✅ `__TAURI__.stronghold` / `__TAURI__.keyring` | ❌ |

**평가**: Kalsae는 이 영역에 대한 프레임워크 레벨 API가 없다. 앱 개발자가 매번
Security.framework / DPAPI / libsecret을 직접 호출해야 하고, JS 측에서는 IPC를 통해
수동으로 노출해야 한다. 보안적으로는 **유연**(앱이 본인의 보안 요구에 맞게 구현)
하지만 **default-safe** 측면에서는 부재.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐ | 내장 API 없음 — 앱이 직접 구현해야 함 |
| Tauri | ⭐⭐⭐⭐ | keyring + stronghold 플러그인 우수 |
| Wails | ⭐ | 내장 없음, 가이드도 거의 없음 |

**Kalsae 개선 필요성**: 🟡 향후 검토 — `KSCredentialStore` 또는 `kalsae-plugin-keyring` 형태로 도입 가능.

### 3.15 영역 14: WebView 프로세스 격리

WebView를 호스트 프로세스와 분리하여 sandbox escape 영향도를 제한.

| 항목 | Kalsae | Tauri (v2) | Wails (v3-alpha) |
|------|--------|-------------|------------------|
| **Windows: WebView2 프로세스 모델** | ✅ WebView2 기본 multi-process — Edge 런타임이 별도 프로세스로 분리 | ✅ 동일 | ✅ 동일 |
| **macOS/iOS: WKWebView Content Process** | ✅ WKWebView 기본 — 별도 WebContent 프로세스 | ✅ 동일 | ✅ 동일 |
| **Linux: WebKitGTK process model** | ✅ WebKitGTK 6.0 multi-process (`WebKitWebProcessExtension` 분리) | ✅ 동일 | ✅ 동일 |
| **JS Isolated World / Content World** | ⚠️ Kalsae는 단일 world (`__ks` injection이 main world) | ✅ Tauri는 `__TAURI_INTERNAL__`을 isolated world에 주입 가능 (옵션) | ❌ |
| **여러 창 간 프로세스 분리** | ⚠️ 동일 앱 내 multi-window는 같은 WebContent 프로세스 풀 공유 (OS 동작) | ⚠️ 동일 | ⚠️ |

**평가**: 프로세스 격리 자체는 **OS WebView 엔진의 책임**이므로 세 프레임워크 모두
유사한 수준. Tauri의 차별점은 **isolated content world**(MV3 스타일) 지원으로,
페이지 JS가 `__TAURI__` 객체를 hijack할 수 없도록 격리할 수 있다.
Kalsae는 현재 main world injection이라 페이지가 `window.__ks`를 변조할 수 있다.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐⭐⭐ | OS 기본 격리 활용. isolated world 미적용 |
| Tauri | ⭐⭐⭐⭐ | Isolated content world 옵션 제공 |
| Wails | ⭐⭐⭐ | OS 기본 격리만 활용 |

**Kalsae 개선 필요성**: 🟡 — WKUserScript `WKContentWorld.defaultClient` /
WebView2 `AddHostObjectToScript` isolated world 옵션 도입 검토.

### 3.16 영역 15: 자동 업데이트 보안

업데이트 패키지 무결성/서명 검증, 다운그레이드 방지, 채널 분리.

| 항목 | Kalsae | Tauri (v2) | Wails (v3-alpha) |
|------|--------|-------------|------------------|
| **업데이트 메커니즘** | ⚠️ **RFC-001에서 설계 중** (구현 전) | ✅ `tauri-plugin-updater` 기본 제공 | ❌ 사용자 책임 |
| **서명 검증** | ⚠️ RFC-001 — Ed25519 또는 minisign 예정 | ✅ Ed25519 (`tauri signer`) | ❌ |
| **무결성 (Checksum)** | ⚠️ RFC-001 예정 | ✅ SHA-256 | ❌ |
| **HTTPS 강제** | ⚠️ RFC-001 예정 | ✅ 강제 | ❌ |
| **다운그레이드 방지** | ⚠️ RFC-001 예정 — semver 비교 명시 필요 | ✅ semver 검증 | ❌ |
| **사용자 동의 UI** | ⚠️ RFC-001 예정 | ✅ 기본 다이얼로그 + 커스텀 가능 | ❌ |

**평가**: Kalsae는 현재 자동 업데이트 메커니즘이 **존재하지 않음**(RFC-001 단계).
Tauri는 업계 모범 수준의 updater를 갖고 있고, Wails는 명시적으로 다루지 않는다.
RFC-001 구현 완료 시 Tauri 수준 도달 가능.

| 프레임워크 | 점수 | 근거 |
|-----------|------|-------|
| Kalsae | ⭐ | 미구현. RFC-001에서 예정 |
| Tauri | ⭐⭐⭐⭐⭐ | Ed25519 + SHA-256 + semver + HTTPS 모두 충족 |
| Wails | ⭐ | 미구현, 사용자 책임 |

**Kalsae 개선 필요성**: 🔴 RFC-001을 우선순위로 진행. 본 RFC와 P3.2 항목이 연결됨.

## 5. Kalsae 개선 로드맵 제안

본 비교 프레임워크를 바탕으로 도출된 우선순위별 개선 제안:

### 5.1 🔴 Phase 1 (Critical — Breaking Changes)

| # | 제안 | 현재 | 제안 | 영향 |
|---|------|------|------|------|
| P1.1 | `commandAllowlist` 의미론 정정 | `nil`(미설정) → "등록된 non-internal 명령 전체 자동 허용"<br>`[]` → 모두 거부 | `nil`을 `[]`와 동일하게 "deny-all"로 재해석<br>**또는** `capabilities` 블록을 단일 진입점으로 승격하고 `commandAllowlist`를 deprecated 처리 | 모든 기존 앱에서 `commandAllowlist` 또는 `capabilities` 명시 필요 |
| P1.2 | `KSNotificationScope` 기본 차단 | `post: true, cancel: true, requestPermission: true` | 모두 `false` (OS 권한 프롬프트가 2차 방어로 남음) | 알림 사용 앱에서 명시적 true 필요 |

**마이그레이션 전략**:
- Swift 컴파일러 경고(Deprecated)를 1~2 마이너 버전 동안 출력
- `Kalsae.json`에 `"schemaVersion": 2` 필드 도입 (v1 호환성 유지)
- `kalsae migrate` CLI 명령으로 자동 변환 지원

### 5.2 🟡 Phase 2 (Non-breaking Enhancements)

| # | 제안 | 설명 |
|---|------|------|
| P2.1 | `$CACHE`, `$CONFIG`, `$DESKTOP`, `$DOWNLOAD`, `$APPCACHE`, `$APPLOG` 플레이스홀더 추가 | macOS: `cachesDirectory`, Linux: XDG_CACHE_HOME, Windows: `FOLDERID_LocalAppData` 등 매핑 |
| P2.2 | 동적 창 생성(`__ks.window.create`)에 CSP/가상호스트 재적용 | 현재 "structural settings only" limitation 해소 |
| P2.3 | 보안 감사 로그 채널 (`KSLogSecurity`) | 모든 scope 거부를 구조화 로그로 기록 |
| P2.4 | `kalsae security check` CLI 명령 | 설정 파일 분석 → 보안 취약점 리포트 생성 |
| P2.5 | `KSHTTPScope`의 위험 헤더 차단 목록 도입 | `Host`, `Cookie`, `Authorization` 등 화이트리스트 외 차단 |

### 5.3 🟢 Phase 3 (Long-term / v2)

| # | 제안 | 설명 |
|---|------|------|
| P3.1 | `capabilities` 시스템을 단일 진입점으로 정착 | `KSCapabilitiesConfig`는 이미 구현 완료 — 남은 작업은 (a) `commandAllowlist` deprecation 경로 (b) Permission identifier 표준 작성 (c) `kalsae init` 템플릿에 capabilities 블록 포함 |
| P3.2 | `kalsae sign` + 검증 도구 | RFC-001 updater와 연계 — Ed25519/minisign 결정 |
| P3.3 | WebView2 / WKWebView Isolated content world 옵션 | `window.__ks` 변조 방지 — 영역 14에서 식별된 갭 해소 |
| P3.4 | `KSCredentialStore` (keyring 통합) | macOS Keychain · Windows DPAPI · Linux libsecret 통합 추상화 — 영역 13 갭 해소 |
| P3.5 | WebView2 전용 보안 기능 (`add_PermissionRequested` 등) | Windows에서 카메라/마이크/위치 권한 통제 |

## 6. 프레임워크 활용법

이 비교 프레임워크는 다음과 같이 활용할 수 있다:

### 6.1 보안 감사 체크리스트

각 영역별로 Kalsae의 현재 상태를 검사하는 체크리스트로 사용:

```markdown
## 보안 감사 체크리스트 (Kalsae 앱 대상)

- [ ] P1.1: `commandAllowlist`가 명시적으로 설정되었는가? (기본 허용에 의존하지 않는가?)
- [ ] P1.2: `notifications.*`가 필요한 경우에만 `true`로 설정되었는가?
- [ ] `fs.allow`가 최소 권한 원칙을 따르는가?
- [ ] `http.allow`가 필요한 API만 허용하는가?
- [ ] `commandRateLimit`이 적절히 설정되었는가? (권장: rate=100, burst=200)
- [ ] `contextMenu`가 앱 UI에 적합하게 설정되었는가?
- [ ] `downloads.enabled`가 불필요하게 `true`로 되어 있지 않은가?
- [ ] `navigation.allow`가 필요한 외부 URL만 허용하는가?
- [ ] `crossOriginIsolation`이 필요할 때만 활성화되었는가?
- [ ] DevTools가 릴리스 빌드에서 비활성화되었는가? (강제되므로 확인 불필요)
```

### 6.2 신규 기능 설계 템플릿

새로운 IPC 명령을 추가할 때 다음 질문을 던지는 템플릿:

```
1. 이 명령이 접근하는 리소스 유형은? (파일/네트워크/셸/UI/기타)
2. 기존 scope 중 어디에 속하는가? (fs / http / shell / ...)
3. 새로운 scope가 필요한가? (기존 scope로 검증 불가능한 리소스인가?)
4. 기본값은 deny인가? (명시적 opt-in이 필요한가?)
5. 거부 시 반환할 에러 코드는 무엇인가?
6. 이 명령이 rate limiting의 대상이 되어야 하는가?
7. 명령 인자에 path/URL이 있다면 → path glob validation이 필요한가?
```

## 7. 미결 사항 및 향후 과제

| # | 항목 | 설명 |
|---|------|------|
| 1 | **iOS/Android PAL 보안 수준** | 현재 preview 상태로, WebView 경계 보안이 데스크톱과 동일한 수준인지 검증 필요 |

| 4 | **Wails v3 stable 시점 재평가** | 본 RFC는 Wails v3-alpha 시점 스냅샷. stable 릴리스 후 영역별 점수 재산정 필요 |
| 5 | **`kalsae security check` CLI** | 위 체크리스트를 자동화한 CLI 명령 — 구체적 설계는 별도 RFC 필요 |
| 6 | **TLS / 인증서 정책** | 커스텀 CA, 자체서명 인증서, 인증서 핀닝 정책을 프레임워크 차원에서 노출할지 — 현재 OS 기본값 의존 |
| 7 | **점수 가중치 부재** | 현재 §4.1은 단순 합산 — IPC/업데이트 보안 같은 critical 영역에 가중치를 두는 방안 검토 |
| 8 | **클립보드 read/write 권한 분리** | 현재 `KSClipboardScope`는 단일 enable 플래그 — Tauri처럼 `allow-read` / `allow-write` 분리 검토 |
| 9 | **커스텀 프로토콜 origin 격리** | `ks://app/` (Linux) vs `https://app.kalsae/` (Windows) 의 origin 동치성 — Service Worker 등록, IndexedDB 분리 동작 차이 검증 필요 |
| 10 | **SBOM / 의존성 감사** | Tauri는 `cargo audit` 통합. Kalsae는 SwiftPM 의존성 트리 + WebView2 NuGet → 통합된 감사 도구 필요 |
